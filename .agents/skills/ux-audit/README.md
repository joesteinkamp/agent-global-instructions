# ux-audit-skill

A Claude Code skill (`/ux-audit`) that audits UI screenshots against 15 published UX
heuristics frameworks and produces severity-rated, principle-cited findings, calibrated
0–100 scores, annotated screenshots, and a self-contained HTML report.

**Usage:** in any Claude Code session with the skill installed, provide 1+ screenshots and
say "audit this design" / "/ux-audit". One round of intake questions (goal, persona,
platform, artifact stage), then the audit runs. Outputs land in `audits/<slug>-<date>/`:
`findings.json`, `summary.md`, `annotated/*.png`, `report.html`.

**Install:** `ln -s ~/projects/ux-audit-skill ~/.claude/skills/ux-audit`

## What goes into a score

A score is never a vibe. Every audit follows the same evidence chain, and each step has a
source-of-truth file in `references/`:

1. **Cite** — every candidate finding must reference ≥1 principle from a published
   framework (the 15 below), drawn from the controlled vocabulary in
   `references/registry.json`. No citation, no finding.
2. **Measure** — every numeric contrast/size claim must come from `scripts/measure.py`,
   not from the eye. Unmeasured suspicion is capped at medium severity and phrased
   "appears low-contrast — verify" (`references/frameworks/wcag.md`, hard rule 2).
3. **Filter** — every candidate passes four gates before it counts
   (`references/filter.md`).
4. **Rate** — each surviving finding gets a severity weight and a reach band
   (`references/scoring.md`).
5. **Score** — findings roll up into six weighted category sub-scores and one overall
   0–100 number, sanity-checked against fixed calibration bands (`references/scoring.md`).

### The 15 frameworks (heuristics)

The skill loads framework files by tier so each audit only pulls in what it needs (Tier 1
= judgeable from a single static screenshot; Tier 2 = needs a multi-screen flow; Tier 3 =
behavioral, **hypothesis-only and excluded from scoring**). Across the active files there
are **110 controlled principle IDs** — the full immutable list lives in
`references/registry.json`, regenerated by `scripts/build_registry.py`.

| Framework | Prefix | Tier | Default category | Source |
|---|---|---|---|---|
| Nielsen's 10 Usability Heuristics | `NLS` | 1, 2 | usability | nngroup.com |
| Norman's Design Principles (affordances, signifiers, gulfs) | `NRM` | 1, 2 | usability | *The Design of Everyday Things* |
| Shneiderman's 8 Golden Rules | `SHN` | 2 | usability | cs.umd.edu/~ben |
| Tognazzini's First Principles of Interaction Design | `TOG` | 1, 2 | usability | asktog.com |
| Recurring failures (named, repo-curated checks) | `RF` | 1, 2 | usability | seeded from Qualia |
| Gestalt Principles of Grouping | `GST` | 1 | visual_layout | grouping principles |
| Gerhardt-Powals' Cognitive Engineering Principles | `GP` | 1 | cognitive_load | heuristic evaluation |
| Hick's Law | `HICK` | 1 | cognitive_load | lawsofux.com |
| Fitts's Law | `FITTS` | 1 | cognitive_load | lawsofux.com |
| Miller's Law | `MILL` | 1 | cognitive_load | lawsofux.com |
| Jakob's Law | `JAKOB` | 1 | cognitive_load | lawsofux.com |
| Peak-End Rule | `PEAK` | 2 | cognitive_load | lawsofux.com |
| WCAG 2.1 (screenshot-auditable subset) | `WCAG` | 1 | accessibility | w3.org/WAI |
| 10 Content Design Heuristics | `CNT` | 1, 2 | content | uxcontent.com |
| Cialdini's Principles of Influence + dark-pattern checks | `CLD` / `DARK` | 1 | trust_persuasion | Cialdini, *Influence* |

> **Excluded from scoring:** the Fogg Behavior Model (`FOGG`, Tier 3) ships as
> `references/frameworks/fogg.md` for background only. It is `status: informational`, kept
> out of the registry, and never loaded during an audit — behavioral/conversion claims
> need analytics or A/B data the skill deliberately refuses to fake.

Each framework file lists its principles, a one-line definition per principle, and
**screenshot-observable symptoms** (the concrete visual cues that justify a citation) plus
a worked JSON example. See `references/frameworks/_format.md` for the file contract.

### Evidence: deterministic measurement

Subjective contrast/size judgments are not allowed to drive numbers. `scripts/measure.py`
(Python stdlib + Pillow, fully offline) does the measuring:

- an auto pass seeds best-effort measurements for all screens, and
- targeted `--region … --probe contrast|size` probes verify any element suspected of a
  WCAG contrast or target-size failure.

It reports `unmeasurable` rather than guessing. The relevant thresholds:
WCAG 1.4.3 text contrast ≥4.5:1 (≥3:1 for large text), 1.4.11 non-text contrast ≥3:1, and
2.5.8 target size ≥24px (44px best practice on declared-touch platforms). Anything
invisible in a static image — focus order, ARIA, alt text, keyboard behavior, real timing
— is explicitly **out of scope** and called out in the report's methodology note.

### The four-gate filter

Every candidate finding must survive all four gates in `references/filter.md` (verbatim
from Qualia's prime directive) or it is dropped silently — it does **not** get demoted to a
low-severity finding:

1. **Blocker check** — does it plausibly impede the user's stated goal?
2. **Standard check** — is this just a conventional pattern? (Don't flag conventions.)
3. **Clutter check** — does the fix add new UI when an existing path already serves?
4. **Steelman check** — is there a strong reason a good designer chose this on purpose?

Cross-framework duplicates merge into one finding with multiple citations; findings are
capped at 5 per category (overflow is logged, never silently truncated); and a clean
screen is a valid result — passing checks are recorded in `clean_checks`.

### Severity, reach, and the 0–100 roll-up

Severity weights and reach bands (verbatim from the Qualia rubric, `references/scoring.md`):

| Severity | Meaning | Weight |
|---|---|---|
| critical | Prevents user goal completion | 4 |
| high | Significant friction on the critical path | 3 |
| medium | Reduces confidence or adds friction | 2 |
| low | Polish or edge case | 1 |

`importance = severityWeight × reachBand`, where reach (screens/elements touched) bands as
≤1 → 1.0 · ≤3 → 1.5 · ≤9 → 2.0 · >9 → 3.0. Findings sort by importance.

Each finding feeds one of **six category sub-scores (0–100)**, combined into the overall
score by these weights:

| Category | Weight |
|---|---|
| usability | 30% |
| cognitive_load | 15% |
| visual_layout | 15% |
| accessibility | 15% |
| content | 15% |
| trust_persuasion | 10% |

The overall score is the weighted average **then sanity-checked against fixed bands** — if
the band description contradicts the math, the band wins and sub-scores are revisited:

- **0–49** fundamentally broken; primary action missing/unreachable/non-functional
- **50–65** broken flows; invisible primary actions; users likely abandon without external guidance
- **66–79** functional, standard design; usable but unremarkable
- **80–89** strong, well-considered with minor gaps
- **90+** exceptional; no blockers, no friction on critical path, clear hierarchy

Score compression (clustering everything 70–89) is treated as a calibration failure: the
full range is used, mediocre designs land 50–65 and strong ones 88–94. Per-category 90/50/20
anchors and the dark-pattern floor (any 2+ `DARK` findings cap trust_persuasion at 35) are
defined in `references/scoring.md`.
